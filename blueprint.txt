## Slide 2: Introduction & Project Context

### Visual Blueprint

* **The Problem Space:** Inefficient manual record-keeping, delayed entry queues, fragmented transaction history.
* **The User Pain Point:** Lack of affordable personalized coaching; static generic fitness apps that ignore specific locally available gym equipment.
* **The Solution Paradigm:** A dual-ecosystem consolidating desktop administrative control via Electron and automated mobile client features paired with context-aware AI recommendations.

### Defense Script

> "Fitness centers often handle member tracking, subscriptions, and payment matching via manual operations, leading to data errors and delayed front-desk queues. Concurrently, gym members want personalized workouts but face cost barriers for personal trainers. Existing mobile solutions offer static routines that fail to recognize the actual equipment physically inside a specific venue. To solve this disconnect, our study bridges administrative business automation with personal fitness customization tailored directly to Carreon Gym's real-time inventory."

## Slide 5: Scope and Limitations

### Visual Blueprint

* **In-Scope Functions (Admin):** Member profiles, active/expired subscription records, manual override logs, S3 payment verification, hardware catalog toggles.
* **In-Scope Functions (Mobile):** Scan-based check-in/out, AWS invoice tracking, chat session history, structured biometric input forms.
* **Explicit System Boundaries:** Distinctly isolated from multi-branch synchronization, hardware computer-vision pose tracking, or real-time camera posture correction. Completely reliant on standard network data pathways.

### Defense Script

> "The scope is intentionally engineered for Carreon Gym's administrative operations, including profile updates, subscription triggers, and client progress dashboards. On the client side, features are bounded by profile capture, QR-driven check-in logs, and token-managed AI interaction threads. We emphasize that the system limits its scope to standard data entry pathways and does not include multi-tenant scaling or camera-based real-time computer vision tracking."

## Slide 7: Research Methodology & Conceptual Framework

### Visual Blueprint

* **Framework Model:** Input -> Process -> Output (IPO).
* **Input Layer:** Biometric profiles, target goals, payment records, equipment status toggles.
* **Process Layer:** AI inference mapping, RESTful verification pipelines, JWT token checks.
* **Output Layer:** Dynamic personalized workout schemas, live administrative financial ledgers.

### Defense Script

> "We structured our research using an Input-Process-Output conceptual framework. The Input stage collects user biometrics, training goals, and equipment statuses. The Process stage handles token authentication, runs database operations via PostgreSQL, and manages inference processing. The Output stage delivers tailored workout plans to members while generating live financial ledger reports for administrators."

## Slide 10: System Architecture & Data Flow

### Visual Blueprint

* **Presentation Layer:** Expo React Native Mobile Client & Electron Admin Dashboard.
* **Application Logic Layer:** Node.js + Express REST API handling endpoints.
* **Data & Inference Storage:** PostgreSQL database storage alongside the Gemini API layer.

### Defense Script

> "The architectural framework runs on a three-tier model. The presentation layer uses Expo React Native for members and Electron for the admin desktop dashboard. The application logic tier runs on Node.js and Express to route data requests.
